# 【第1轮·开篇立论】

## 我方立场：知识库的自我进化，本质是"结构化反事实推理"的持续积累

---

### 一、先厘清这场辩论的赛道

在讨论"能自我进化的知识库"时，存在两种截然不同的设计哲学：

| 路线A：**数据驱动的统计积累** | 路线B：**因果结构的反事实推理** |
|---|---|
| 收集更多实验结果 → 归纳规律 | 理解"为什么有效" → 推断边界条件 |
| 进化 = 数据量增长 | 进化 = 解释力增长 |
| 优化已知空间 | 扩展可知空间 |

**我方主张路线B**，并认为在 GPU Kernel Design 这个应用场景下，路线A会导致"高速原地踏步"的系统性失败。

---

### 二、核心主张：三层命题

**命题一：GPU Kernel 优化的知识本质是"条件依赖的因果图"，而非"参数-性能的映射表"**

一个 kernel 的性能由 memory hierarchy、warp occupancy、instruction-level parallelism 的交互决定。这些变量之间存在**非线性因果依赖**：

```
thread block size → shared memory pressure → L1 hit rate → 实际带宽利用率
                ↘ register spilling → instruction stall cycles
```

如果知识库只记录"block_size=256 在 A100 上比 128 快 23%"，它在 H100 或不同 memory access pattern 下会**系统性失效**。

**命题二：自我进化的充分必要条件是"可证伪的知识单元"**

知识库能否进化，取决于它的每一条知识是否携带**可被新实验推翻的假设**。

- 不可进化的知识："tiling size = 64 效果好"（无条件、无边界）
- 可进化的知识："当 arithmetic intensity > 4 FLOP/Byte 且 L2 cache > 40MB 时，tiling size = 64 最优；否则此结论无效"

后者的每个前提条件都是一个**可检验的、可被反事实推翻的命题**。

**命题三：进化的驱动引擎不是"更多数据"，而是"结构化的异常检测"**

真正的自我进化发生在：
1. 新实验结果**违反了现有知识的预测**
2. 系统能**定位是哪条假设链断裂**
3. 系统能**生成最小化修正**而非推倒重来

这要求知识库具备**可追溯的推理结构**，而不只是存储结果。

---

### 三、针对 GPU Kernel Design 的框架锁定

基于以上三个命题，我方提出本辩论的核心框架：

```
自我进化知识库 = 
    知识表示层（因果图 + 条件边界）
  × 实验设计层（反事实探索策略）
  × 异常融合层（假设修正机制）
```

这三层缺一不可。单独优化任何一层都是局部解。

**我方将在后续五轮中逐层展开：**
- 第2轮：知识表示——如何编码 GPU 性能知识的因果结构
- 第3轮：实验设计——如何主动生成"最有价值的反例"
- 第4轮：异常融合——新数据如何触发精确的知识修正
- 第5轮：系统边界——什么是这个框架无法处理的，以及如何应对
- 第6轮：综合——与路线A的正面交锋与最终裁决

---

### 四、一句话锁定赛道

> **这场辩论的胜负标准是：谁的设计能在"从未见过的 GPU 架构"上，用最少的新实验，做出最可靠的优化决策。**

统计积累在这个标准下会输，因为它的知识不迁移。
结构化因果推理会赢，因为它的知识**知道自己的边界**。
