# 【第1轮：开篇立论】

## 我方立场：**以"可证伪的假说单元"为核心，构建假说驱动的自进化知识库**

---

### 一、先定赛道：这场辩论的核心争点是什么

在开始论证之前，我必须先说清楚"这场辩论应该按什么比"。

本题问的是"自我进化的知识库"，但"进化"这个词有两种根本不同的理解：

- **理解A（被动积累式）**：系统运行 → 产生性能数据 → 数据驱动知识更新 → 知识库增长。这是一种"数据驱动的记录系统"。

- **理解B（主动假说式）**：系统提出假说 → 设计实验验证 → 根据证伪结果修订假说 → 知识库进化。这是一种"科学方法论驱动的推理系统"。

**我方的核心主张是：只有理解B才是真正意义上的自进化。理解A本质上是一个精致的日志系统，而非进化系统。**

判断标准我提前亮出：**一个能"自我进化"的知识库，必须具备主动减少自身不确定性的能力，而不仅仅是被动积累观测结果。**

---

### 二、核心方案：假说驱动的知识进化架构（Hypothesis-Driven Evolution）

我方的方案以一个核心数据结构为轴心：

**每一个 Knowledge Unit = 一个可证伪的假说（Falsifiable Hypothesis）**

具体定义如下：

```
Knowledge Unit {
  UUID: <唯一标识>
  Hypothesis: "在 [条件C] 下，对 Kernel K 应用优化策略 S，
               预期可将 [指标M] 提升 [量化范围 δ±ε]"
  Evidence: [实验记录列表]
  Confidence: 基于 Evidence 计算的置信度
  Applicability_Scope: 该假说适用的 Kernel 特征空间
  Status: Created / Revised / Merged / Split / Superseded / Archived
  Parent_UUID: <来自哪张卡片>
  Falsification_Condition: "当观测到 [X] 时，本假说应被修订或废弃"
}
```

**关键设计点：** 每张卡片不只是记录"某策略有效"，而是记录"在什么条件下有效、预期效果是什么、如何才能证伪它"。这使得每张卡片天然具有**可验证性**和**可被超越性**。

---

### 三、进化机制：如何触发状态转换

我方的进化触发链如下：

**触发层（Trigger Layer）**：
1. 新的 Kernel 优化实验完成 → 产生性能数据
2. 数据与某张卡片的预期（δ±ε）对比
3. 若观测值落在置信区间内 → 更新 Confidence（Evidence 追加）
4. 若观测值显著偏离预期 → 触发 Falsification → 启动 Revision 流程

**状态转换规则**：
- `Revised`：假说条件边界被收窄或扩大，CoW 产生新卡片
- `Split`：发现假说在不同 Kernel 类型上行为不一致，拆分为两张范围更精确的卡片
- `Merged`：两张卡片的 Applicability_Scope 有重叠且结论一致，合并为更一般化的卡片
- `Superseded`：新卡片在所有维度上覆盖并超越旧卡片
- `Archived`：连续N次实验无法产生 real delta，或被 Superseded

**核心优势**：状态转换不是人工触发，而是由**假说与观测的偏差**自动触发，这才是"自进化"的真实含义。

---

### 四、反馈机制：如何确保净进化而非净退化

这是本题最难的部分，我方的设计分三层：

**第一层：局部验证（Ablation Study 的轻量化替代）**

由于 Canary 代价高昂，不能频繁触发，我方引入**影子实验（Shadow Experiment）**机制：

- 每次 Revision 或 Merge 操作，系统自动在**当前 Applicability_Scope 内**挑选代表性的 micro-benchmark（不是完整 Canary），运行针对该卡片变化的最小化消融实验
- 目标不是全局验证，而是验证"这次 CoW 操作没有引入局部退化"
- 成本控制：micro-benchmark 的规模由 Applicability_Scope 的大小自适应决定

**第二层：全局一致性检查（Canary 作为长期信号）**

- 每隔固定周期（或累积 K 次 Revision 后），触发一次 Canary
- Canary 结果不直接回滚某张卡片，而是触发**全局归因分析**：哪些 Knowledge Unit 的变化与 Canary 指标的变动最相关
- 这是一个反向追溯机制，而非实时反馈机制

**第三层：不确定性预算（Uncertainty Budget）**

- 系统维护一个全局的 Uncertainty Budget：总知识量受限时，优先保留置信度高的卡片
- 当知识库规模超过上限时，Archived 状态的卡片按置信度排序，低置信度的先被压缩

---

### 五、为什么这是"赛道锁定"

我方的核心框架确立了以下判断标准，这将贯穿后续五轮：

1. **可证伪性**：一个 Knowledge Unit 如果无法说明"在什么情况下自己是错的"，它就不是真正的知识，只是观测记录
2. **主动性**：进化系统必须能主动设计实验，而不只是等待数据
3. **净进化可验证性**：每次操作必须有机制证明自己没有引入退化

**我向对方提出的核心挑战：**

> 如果你的方案没有"可证伪性"这一设计，你如何区分"知识在增长"和"错误在积累"？
> 如果你的方案依赖 Canary 作为主要反馈，你如何处理 Canary 不敏感窗口期内的退化风险？

---

以上是我方第一轮的完整立论。立场明确：**假说驱动的自进化知识库，以可证伪的 Knowledge Unit 为核心，以假说-验证-修订循环为进化引擎。**
