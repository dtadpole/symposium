# 第1轮：开篇立论

## 我方立场：**效用驱动的贝叶斯演化框架**

---

## 一、赛道锁定：这场辩论比什么？

这场辩论的核心不是"能不能存、能不能查"，而是：

> **知识库有没有一个可验证的机制，保证它随时间变得更好——而不只是变得更大。**

我方认为，评判标准应聚焦于两个维度：
1. **演化操作的触发是否有客观依据**（不是人工拍脑袋）
2. **每次演化后是否有可量化的净收益证明**（不是凭感觉说"变好了"）

任何方案，如果无法回答"你怎么知道这次演化是进化而不是退化"，就不是一个完整的方案。

---

## 二、演化机制：事件驱动 × 阈值判断 × 原子操作

### 核心主张：状态转换必须由**可观测事件**触发，由**量化阈值**判断，由**原子操作**执行。

### 触发条件（Trigger）

我方将触发源分为三类：

| 触发类型 | 具体事件 | 目标状态 |
|---|---|---|
| **外部输入触发** | 新信息与现有卡片语义重叠度 > θ₁ | Revised / Merged |
| **使用反馈触发** | 某卡片连续 N 次查询后未被采纳（命中但无效） | Superseded / Archived |
| **结构压力触发** | 总卡片数逼近容量上限 C_max | Split / Archive 候选池激活 |

### 判断标准（Decision Criterion）

每个候选演化操作，必须通过**效用差分检验**才能执行：

```
ΔU = U(KB_after) - U(KB_before) > 0
```

其中效用函数 U(KB) 定义为：

```
U(KB) = Σᵢ [ relevance(kᵢ) × utility(kᵢ) × recency_weight(kᵢ) ] - λ × |KB|
```

- `relevance`：卡片与当前查询分布的语义匹配度
- `utility`：历史上该卡片被采纳的成功率
- `recency_weight`：时间衰减权重（越旧越低）
- `λ × |KB|`：**容量惩罚项**，这是资源受限的数学表达

**只有 ΔU > 0，演化操作才被批准执行。**

### 执行流程（Execution）

所有操作遵循预设假设的 Copy-on-Write + Append-only 原则：

1. 计算 ΔU（预演算）
2. ΔU > 0 → 生成新卡片（新 UUID），更新状态标记
3. 旧卡片标记为 Superseded / Archived，**永不删除，保留可回溯**
4. 写入操作日志（包含触发事件、ΔU 值、执行时间戳）

## 三、反馈机制：净进化的数学保证

### 核心主张：**用贝叶斯更新 + 效用单调性约束，从数学上锁死退化路径。**

### 问题一：如何数学保证知识库变得更切合目标？

我方引入**知识库全局效用 U(KB, t)**，并要求系统维护一个**单调不降约束**：

```
U(KB, t+1) ≥ U(KB, t)   ∀t
```

实现机制：**每次演化操作前，系统必须模拟执行后的 U 值，低于当前值则拒绝操作。**

这不是事后评估，而是**事前门控（pre-execution gate）**。

### 问题二：用什么指标衡量知识库在变好还是变坏？

我方提出三层指标体系：

| 层次 | 指标 | 计算方式 |
|---|---|---|
| **精准度** | Query Satisfaction Rate (QSR) | 查询命中且被采纳 / 总查询次数 |
| **覆盖度** | Coverage Entropy (CE) | 当前 KB 对目标领域的语义覆盖熵 |
| **健康度** | Active Ratio (AR) | Active 状态卡片 / 总卡片数 |

三项指标加权合成 U(KB)，每轮演化后自动更新。

### 问题三：资源受限时，如何决定留、合并、废弃？

我方方案：**效用排序 + 边际收益竞争**

当容量压力触发时，系统执行以下流程：

```
1. 计算所有 Active 卡片的边际效用 MU(kᵢ) = ∂U/∂kᵢ
2. 按 MU 升序排列，末尾候选进入"淘汰池"
3. 对淘汰池中的卡片，检查是否可 Merge（语义相似度 > θ₂）
   → 可 Merge：合并为新卡片，两旧卡片标记 Merged
   → 不可 Merge：效用最低者标记 Archived
4. 执行后重算 U(KB)，确认 ΔU > 0
```

**关键保证：Archived 卡片不删除，仅退出 Active 计算池。**
这意味着知识不会真正消失，只是被"冷冻"——当未来查询需要时，仍可被召回激活。

---

## 四、框架总结：为什么这是更优赛道？

我方方案的核心优势在于：

**它不只描述"怎么做"，还给出"怎么验证做对了"。**

- 演化操作有客观触发条件，不靠人工干预
- 每次操作前有效用门控，数学上拒绝退化
- 资源受限时有明确优先级算法，不凭感觉取舍
- 旧知识以 Archived 形式保留，进化可逆、历史可溯

我方挑战对方在立论时必须回答：**你的方案如何从数学上证明每次演化是净进化？** 如果没有这个答案，方案再精巧，也只是一个"能动的知识库"，而不是一个"会进化的知识库"。
