**【第1轮：开篇立论】**

---

## 一、明确立场

我方的核心主张是：

> **知识库的自我进化，应以"可验证的因果理解"为核心驱动力，而非以性能数字为直接进化信号。**

具体方案名称：**Hypothesis-Driven Causal Evolution（HDCE）**

这个方案的核心思路是：每一次知识单元的状态转换，必须以"已验证或已反驳的因果假设"为前提，而不是以"性能提升了X%"为充分条件。没有因果理解的性能提升，不构成可复用的知识，只是噪声。

---

## 二、关键定义

**Knowledge Unit（KU）**：一个三元组 `(H, E, C)`：
- `H`：Hypothesis——这个优化为什么有效的因果命题（可被证伪）
- `E`：Evidence Set——支撑或反驳 H 的实验记录（含 delta、context、cost）
- `C`：Confidence Score——基于 E 的贝叶斯置信度，区间 [0,1]

**净进化（Net Evolution）**：一次状态转换操作，使得知识库的全局期望性能 `E[Δperf | KU_set]` 严格提升，同时知识库的平均 Confidence 不下降。

**无效实验（Null Experiment）**：实验结果显示 `|Δperf| < ε`（ε 为预定显著性阈值），即 Hypothesis 未被证实也未被反驳，属于弱信号。

---

## 三、判断标准

本场辩论应按以下标准评判哪个方案更好：

1. **验证可靠性**：方案是否能区分"真实进化"与"噪声涨跌"？
2. **资源效率**：在 Canary 代价高、计算资源有限的前提下，实验预算的使用是否最优？
3. **可复用性**：积累的知识能否迁移到新的 Kernel 场景，还是只适用于当前 benchmark？
4. **可解释性**：系统能否说明"为什么这个 KU 触发了进化"，而不是黑箱决策？

---

## 四、两个核心论点

### 论点一：性能数字不能直接驱动进化，因果假设才是可靠信号

在 GPU Kernel 优化中，`Δperf` 受硬件状态、编译器版本、输入规模等多重噪声因素影响。如果直接以性能数字触发 KU 的状态转换，会导致：

- **假阳性进化**：噪声导致的短暂提升被误记为有效策略
- **知识退化**：下游系统复用无效 KU，在新场景中性能反而下降

我方的解决方案是：**每个 KU 必须携带一个可证伪的 Hypothesis `H`，进化触发条件不是"性能变好了"，而是"H 被实验证实或证伪"。**

具体触发机制：
```
触发条件：新实验 e_i 产生后，
  若 Bayesian_update(C | e_i) 使得 C 越过阈值 τ_confirm（默认 0.85），则触发 Revised 或 Confirmed
  若 C 越过阈值 τ_reject（默认 0.15），则触发 Superseded 或 Archived
  若 |Δperf| < ε 且 C 无显著变化，则记录为 Null Experiment，KU 状态不变
```

无效实验的处理：Null Experiment 不是废物。它作为 `E` 的一部分写入 KU，降低未来对该 H 的重复实验优先级（避免资源浪费），同时更新"该 Hypothesis 的可证伪性评分"——如果一个 H 反复产生 Null，说明 H 本身粒度太粗，触发 Split 操作，将 H 分解为更细粒度的子假设。

---

### 论点二：主动实验设计应以"信息增益/成本"比为排序标准，并设置递归深度上限

当系统对某个 KU 的理解深度不足时（`C` 处于模糊区间，如 [0.3, 0.7]），系统应主动设计实验。但主动实验有成本，必须有优先级机制。

我方提出 **EIG/Cost 排序框架**：

```
对候选实验 x_j，计算：
  EIG(x_j) = H(C_before) - E[H(C_after | x_j)]  // 期望信息增益（熵减少量）
  Cost(x_j) = compute_time + memory + opportunity_cost
  Priority(x_j) = EIG(x_j) / Cost(x_j)
```

系统在资源预算 `B` 内，贪心选取 Priority 最高的实验序列执行。

关于递归子假设的管理：
- 允许一个 H 分解为子假设 `{h_1, h_2, ..., h_k}`，最多递归 2 层（防止爆炸）
- 子假设的 Priority 继承父假设的权重折扣：`Priority(h_i) = Priority(H) × λ^depth`，`λ=0.6`
- No-Regret 实验的判断标准：`Cost(x_j) < C_threshold` 且 `EIG(x_j) > EIG_min`，无论结果如何都值得做
- High-cost/uncertain 实验需通过"实验委员会评分"（多个 KU 的关联 Confidence 综合投票）才能触发

---

## 五、核心焦点

我认为本场真正的争议集中在以下两点：

**焦点一：进化的触发单元是"性能信号"还是"因果假设验证"？**
这是两种根本不同的系统哲学。对方可能主张以性能为直接驱动，我方认为这在噪声环境下不可靠，必须以因果理解为中介。

**焦点二：在资源受限下，系统应"保守验证"还是"主动探索"？**
Canary 代价高意味着不能频繁全局验证。我方的答案是：用低成本的局部实验（EIG/Cost 优先）填充验证空白，只在积累足够局部信号后才触发 Canary，而不是等待 Canary 来告诉你该不该进化。
