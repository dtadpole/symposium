## 一、各方立场总结

### Claude 的核心主张

Claude 主张以**任务效用（Task Utility）为唯一合法演化信号**，通过可形式化的效用函数驱动知识库状态转换，并以严格的净效用门槛（MUR原则）约束每次操作。其框架的核心特征是：演化触发条件完全锚定在下游任务完成质量上，内容结构分析仅作为效用估计的辅助手段，而非第一信号；冷启动问题通过贝叶斯先验（邻域效用传播）处理，资源约束通过边际效用置换解决。

### ChatGPT 的核心主张

ChatGPT 主张以**效用密度（Utility Density）为统一目标函数**，通过四步法（信号采集→候选评估→效用预演→条件提交）驱动演化，并以影子评估层作为净进化的操作性保证。其框架强调三个可量化信号（命中率HR、语义重叠度SO、内部一致性张力IT），以及动态权重调整使目标函数本身可随任务漂移而演化。

### 双方的根本分歧

双方的分歧在表面上是"内容驱动 vs. 效用驱动"，但更深层的分歧是：

**演化决策的前瞻性 vs. 后验性**——Claude 坚持演化决策必须基于任务效用的历史观测，ChatGPT 的影子评估试图在操作前预演效用变化。这带来了一个真正的工程张力：Claude 方案在冷启动时依赖贝叶斯先验，ChatGPT 方案在新卡片无历史数据时依赖语义相似度估计，**两者在最困难的场景下都不得不借助内容信号**，但双方都声称自己是纯效用驱动。

这个矛盾是本场辩论未被彻底解决的核心张力。

## 三、核心争点裁决

### 争点1（演化机制）：哪方的方案更具体、更可行？

**裁决：Claude 胜，但优势有限。**

Claude 对演化机制的形式化更为严谨：六种状态转换的触发条件有可操作的数值定义，Jaccard重叠度针对查询集（行为信号）而非内容集的区分，以及Split触发条件在被质询后的即时修正（加入聚类清晰度条件），均体现了工程层面的严密性。

ChatGPT的四步法框架在逻辑结构上是清晰的，但关键缺陷在于：影子评估的计算复杂度（Claude指出的$O(N^2)$候选问题）未被回应；HR和IV两个核心信号中，IV的计算路径始终含糊。

然而两方都面临同一个实质性困境：**任何演化操作在缺乏历史数据时，都必须借助内容语义作为效用估计的代理**。双方的分歧更多是在"是否承认这一点"上，而非在机制设计上有根本差异。Claude更诚实地承认并形式化了这一依赖（贝叶斯先验），ChatGPT则在名义上坚持"纯效用驱动"但实际上在同样的地方引入了语义相似度估计。

### 争点2（反馈机制）：哪方的方案更能从数学/工程上保证净进化？

**裁决：Claude 胜，优势较为明显。**

Claude 的净进化保证有三个层次：MUR的点估计比较 → 置信区间扩展（下置信区间 > 上置信区间）→ 误差有界条件下的概率保证形式化。这是本场唯一一个将"保证的可靠性本身"纳入分析的论证。

ChatGPT的净进化保证停留在"每次提交前验证ΔU > 0"，Claude在第3轮正确指出这是循环论证：保证的质量完全取决于预演的准确率，而预演准确率本身未被分析。ChatGPT在第4轮的回应（"外部锚点打破内部循环"）是有价值的补充，但仍未给出当预演误差为$\delta$时退化概率的量化分析，而Claude已就此给出了$1 - F(\delta / \Delta U_{min})$的框架。

动态权重$w_i(t)$是ChatGPT第4轮的亮点，但这个机制本身的收敛性和稳定性未被分析——权重自适应会不会引入振荡？收敛到什么？这些问题在限时辩论中未被展开，形成了论证的空白。

## 五、综合建议：最优设计方案

综合双方最强的论点，一个最优的自我进化知识库设计应当是：

### 核心架构

**不可变 KU + 候选变更包 + 双信号触发 + 分层效用估计 + 置信度约束决策**

### 具体方案

**第一层：双信号触发（吸收两方优点）**

演化操作的候选生成，同时接受两类信号：
- **内容结构信号**（ChatGPT的HR、SO、IT）：快速、低成本，用于生成候选操作集，不直接触发执行
- **任务效用信号**（Claude的查询集效用历史）：准确、有延迟，用于对候选操作集进行效用差分验证

这两类信号的分工是：内容信号是**候选生成器**，效用信号是**准入裁判**。两者的角色不同，而非互相替代。这既解决了效用信号在冷启动场景下的数据稀疏问题（依赖内容信号生成候选），又保证了最终准入决策锚定在任务目标上。

**第二层：分层效用估计（解决冷启动）**

新卡片的效用估计采用Claude的贝叶斯框架，但先验的来源进一步结构化为三层：
1. 邻域效用传播（Claude方案）：语义近邻的历史效用均值
2. 主题类别统计：该卡片所属领域的历史平均效用
3. 来源可靠性权重：信息来源的历史表现

先验权重随真实查询观测数量衰减，避免先验主导成熟卡片的效用评估。

**第三层：置信度约束决策（解决循环论证）**

# 最终综合答案：自我进化知识库的完整设计方案

## 一、核心分歧的裁决

五轮辩论的根本分歧只有一个：

> **进化好坏的最终裁判，是内部结构一致性，还是外部目标适应度？**

**裁决：两者都必须有，但层级不同。**

内部结构一致性是**约束条件**（Constraint）——违反它的演化操作不被允许执行。

外部目标适应度是**优化目标**（Objective）——系统演化的方向由它决定。

这两者的混淆，是双方各自方案最深层的缺陷：Claude 方过度强调效用信号，忽视了内部一致性作为硬约束的必要性；ChatGPT 方将内部结构质量与外部效用并列，模糊了优化方向。

## 二、基础定义

### Knowledge Unit（KU）的完整表示

```
KU = {
  id:          UUID（不可变）
  content:     结构化命题（不可变，CoW 保证）
  state:       Created | Active | Revised | Merged | Split | Superseded | Archived
  confidence:  P(KU 有效 | 历史任务证据)  ← 贝叶斯后验，可变
  usage_log:   [(query_id, outcome, timestamp), ...]  ← 效用历史，追加
  relations:   [(KU_id, relation_type), ...]  ← 有向关系图
  lineage:     [parent_id, ...]  ← 演化溯源链
  cost:        归一化存储/计算成本
}
```

### 知识库全局状态

```
KB(τ) = { KU₁, KU₂, ..., KUₙ }
约束：Σ cost(KUᵢ) ≤ C_max
```

### 系统目标函数（唯一裁判）

```
Fitness(KB, τ) = E_{q ~ T(τ)} [ Quality(KB, q) ] / Σ cost(KUᵢ)
```

其中：
- `T(τ)` 是时刻 τ 的目标任务分布（允许随时间漂移）
- `Quality(KB, q)` 是知识库对查询 q 的任务完成质量（由下游反馈实测）
- 分母的成本项施加资源压力，防止效用因堆砌而虚增

**关键设计**：`T(τ)` 不是固定的。它由真实查询日志的滑动窗口估计，每隔固定周期更新。目标函数本身是动态的，这解决了"目标漂移"的担忧——系统优化的是当前目标分布，而非历史快照。

## 三、演化机制：六种状态转换的完整规格

### 总体流程（四层架构）

```
Layer 1: 信号采集     → 持续记录每个 KU 的使用日志
Layer 2: 候选生成     → 根据阈值触发候选操作（非立即执行）
Layer 3: 预提交验证   → 在内存沙盒中计算 ΔFitness
Layer 4: 条件提交     → ΔFitness > 0 则写入，否则撤销
```

**Layer 3 是整个系统的核心防退化机制**：任何演化操作在真正写入之前，必须先在内存中模拟执行，对最近 N 次真实查询重跑检索，确认 Fitness 不下降才提交。

---

### 六种状态转换规格

#### Created → Active

**触发条件**：新知识单元被注入系统

**判断标准（双重门槛）**：

```
准入条件 1（内部约束）：
  新 KU 与现有库无逻辑矛盾
  即：∀ KUⱼ ∈ KB，¬Contradict(KU_new, KUⱼ)

准入条件 2（外部效用）：
  U_prior(KU_new) > U(KU_worst)
  其中 U_prior 由贝叶斯邻域先验计算（见下文冷启动处理）
```

两个条件必须同时满足，缺一不可。条件 1 是硬约束（违反则直接拒绝），条件 2 是效用门槛（保证新 KU 至少不比最差的现有 KU 差）。

**冷启动时的先验估计**：

```
U_prior(KU_new) = w₁ · Ū_neighbor + w₂ · U_topic + w₃ · U_source
```

- `Ū_neighbor`：嵌入空间 K 近邻的历史效用均值
- `U_topic`：所属主题类别的统计均值
- `U_source`：信息来源的可靠性权重

先验权重随真实观测积累严格衰减：`α(t) = α₀ / (1 + λ·n_obs(t))`

决策使用置信区间比较而非点估计：

```
接受 KU_new ⟺ U_prior(KU_new) - CI₉₅(KU_new) > U(KU_worst) + CI₉₅(KU_worst)
```

这保证冷启动时的保守行为——新 KU 只有在统计上显著优于最差 KU 时才被接受。

---

#### Active → Revised

**触发条件**：

```
confidence(KUᵢ) 在滑动窗口内下降超过 Δ_conf
且 存在新证据可以修正内容
```

**判断标准**：

```
贝叶斯更新：
  P(KUᵢ 有效 | 新证据) = P(新证据 | KUᵢ 有效) · P(KUᵢ 有效) / P(新证据)

若更新后 confidence < θ_revise，进入修订流程
```

**执行流程**：
1. CoW 生成新 KU'，内容包含修订
2. 新旧 KU 并行运行 A/B 测试（服务于不同查询子集）
3. 当统计显著性达到 p < 0.05 且 Fitness(KB with KU') > Fitness(KB with KU)，KU 变为 Superseded，KU' 变为 Active

---

#### Active → Merged

**触发条件（功能重叠，而非语义相似）**：

```
Jaccard(Qᵢ, Qⱼ) > τ_merge
```

其中 `Qᵢ = { q ∈ Q_history : KUᵢ ∈ Retrieve(q) }` 是**查询集**（行为信号），不是语义向量（内容信号）。

这个设计解决了 Claude 方反复强调的一个正确问题：语义相似 ≠ 功能重叠。两张语义相近的卡片可能服务于完全不同的查询意图；只有查询集重叠才真正意味着功能冗余。

**判断标准**：

```
预提交沙盒验证：
  Fitness(KB_merged) > Fitness(KB_before)
  且 merged KU 无内部逻辑矛盾
```

**执行流程**：
1. 生成候选 merged KU（CoW）
2. 在沙盒中对历史查询重跑，对比 Fitness
3. 满足条件则提交，原两张 KU 变为 Superseded

---

#### Active → Split

**触发条件（双重条件，缺一不可）**：

```
条件 1：Var[u(KUᵢ)] > σ²_split （效用方差高）
条件 2：查询集 Qᵢ 的聚类轮廓系数 > γ_split （存在清晰的使用模式分化）
```

**判断标准**：

```
Fitness(KB with KU_A + KU_B) > Fitness(KB with KU_original) + ε
```

#### Active → Superseded

由 Revised 或 Merged 流程驱动，无独立触发逻辑。

旧 KU 进入 Superseded 状态后，保留在冷存储中（lineage 溯源用），不占用 C_max 的活跃预算。

---

#### Active → Archived

**触发条件**：

```
边际效用 MV(KUᵢ) < θ_archive
且 最近访问时间 > T_decay
```

其中边际效用的精确定义：

```
MV(KUᵢ) = Fitness(KB, τ) - Fitness(KB \ {KUᵢ}, τ)
```

即：删除这张卡片对全库 Fitness 的影响。

**重要推论**：若某 KU 的 MV < 0（删除它反而提升 Fitness），则**不等待存储压力，立即归档**。这是主动去噪，不是被动淘汰。

## 二、关键定义

### 2.1 Knowledge Unit（KU）

一个 KU 是一个四元组：

```
KU = (c, r, u, t)
```

- **c**：内容（content）——结构化的命题、规则或模式
- **r**：关联集（relation set）——与其他 KU 的有向边（支持、冲突、派生、泛化）
- **u**：效用值（utility score）——对当前目标集的贡献度，∈ [0, 1]
- **t**：时间戳向量（time vector）——记录创建、最近访问、最近修订时间

### 2.2 知识库状态

知识库在时间 τ 的状态定义为：

```
KB(τ) = { KU₁, KU₂, ..., KUₙ } 且 Σ cost(KUᵢ) ≤ C_max
```

其中 `cost(KU)` 是存储/计算成本，`C_max` 是资源上限。

### 2.3 "自我进化"的操作定义

知识库的演化是一个**有约束的状态转换序列**：

```
KB(τ) → KB(τ+1)
```

当且仅当满足：`U(KB(τ+1)) > U(KB(τ))` 时，该转换才被允许执行。

其中总效用函数定义为（后文详述）。

## 四、两个核心论点

### 论点一：演化机制必须以"效用差分"为触发条件，而非以"新信息到来"为触发条件

**问题所在：** 绝大多数知识库设计把"有新数据进来"作为更新触发器，这会导致知识库随时间膨胀、退化或震荡。

**我方主张的机制——效用差分触发（Utility-Delta Trigger）：**

每个 KU 的状态转换，必须先计算效用差分 ΔU 才能执行。

定义单个 KU 的效用函数：

```
u(KUᵢ) = α · relevance(KUᵢ, G) 
        + β · confidence(KUᵢ) 
        - γ · redundancy(KUᵢ, KB) 
        - δ · cost(KUᵢ)
```

其中：
- `relevance(KUᵢ, G)`：KU 对目标集 G 的覆盖度——用 KU 在历史查询中被命中且有效解决问题的频率衡量，∈ [0, 1]
- `confidence(KUᵢ)`：KU 的可信度——用贝叶斯后验概率 P(KUᵢ is correct | evidence) 衡量
- `redundancy(KUᵢ, KB)`：KU 与现有库的冗余度——用语义相似度 sim(KUᵢ, KBⱼ) 的最大值衡量，∈ [0, 1]
- `cost(KUᵢ)`：归一化后的资源占用

**六种状态的触发条件（工程层面）：**

| 状态转换 | 触发条件 | 执行动作 |
|---|---|---|
| **Created** | 新 KU 的 u(KUᵢ) > θ_create（创建阈值） | 加入 KB，更新关联图 |
| **Revised** | 新证据使 confidence 下降 > δ_conf | 局部更新 c 和 confidence，重算 u |
| **Merged** | 两 KU 的 sim(KUᵢ, KUⱼ) > θ_merge 且合并后 u 增加 | 生成新 KU，删除原两者 |
| **Split** | 单 KU 对多个不同目标的 relevance 差异 > θ_split | 分裂为两个更专精的 KU |
| **Superseded** | 新 KU 的 u > 旧 KU 的 u 且两者 relevance 高度重叠 | 旧 KU 进入 Archived 状态 |
| **Archived** | u(KUᵢ) < θ_archive 且 最近访问时间 > T_decay | 移出活跃 KB，保留为冷存储 |

**关键约束**：任何转换执行前，必须满足：

```
U(KB_after) - U(KB_before) > 0
```

其中 `U(KB) = Σ u(KUᵢ)` 为全库总效用。若差分为负，操作被拒绝。

### 一、演化机制：状态转换的触发逻辑

我方将每张 Knowledge Unit（KU）的生命周期状态转换建立在**三个可量化信号**之上：

**信号1：查询命中率（Hit Rate, HR）**
每次 KU 被检索系统召回并被用户采纳（采纳 = 产生了后续动作或正反馈），记为一次有效命中。命中率低于阈值 θ_HR，触发 Archived 候选。

**信号2：语义重叠度（Semantic Overlap, SO）**
两张 KU 之间若余弦相似度 > θ_SO，且所覆盖的查询集高度重合，触发 Merged 候选。

**信号3：内部一致性张力（Internal Tension, IT）**
若一张 KU 内部存在相互矛盾的命题（通过逻辑推理图检测），触发 Split 候选。

**状态转换的执行流程（四步法）：**

1. **信号采集层**：持续监控每张 KU 的 HR、SO、IT 三项指标
2. **候选评估层**：当任一指标越过阈值，生成候选演化操作（非立即执行）
3. **效用预演层**：在执行前模拟演化结果，计算 ΔU = U(K_after) - U(K_before)
4. **条件提交层**：仅当 ΔU ≥ 0 时，执行 Copy-on-Write 写入新 KU，原 KU 状态变更

这个四步法的关键在于**第3步"效用预演"**——演化操作不是触发就执行，而是先算收益再决定。

### 二、反馈机制：如何数学保证净进化

这是我方方案最核心的部分。

**定义知识库的效用密度：**

```
U(K) = Σ_i [ HR(i) × IV(i) ] / |K_active|
```

其中：
- `HR(i)` = KU_i 的查询命中率（经验频率）
- `IV(i)` = KU_i 的信息价值（由 KU 覆盖的查询集的熵减量衡量）
- `|K_active|` = 当前处于 Active 状态的 KU 总数（资源约束项）

**这个函数的意义：**
- 分子衡量知识库的"总有效产出"
- 分母施加"资源压力"——越多的 Active KU，密度越低
- 资源受限时，系统自动偏向精简高价值 KU，而非堆砌低质 KU

**净进化保证：**

每次演化操作前，系统必须证明：

```
U(K_after) ≥ U(K_before)
```

若一次 Merged 操作将两张低命中率 KU 合并为一张高命中率 KU，分子上升、分母下降，U(K) 严格递增——这是**可证明的净进化**。

若一次 Archived 操作将一张 HR ≈ 0 的 KU 移出 Active 集，分子不变（它本来就不贡献）、分母下降，U(K) 递增——同样是净进化。

**资源受限时的优先级决策：**

当 `|K_active|` 逼近上限时，系统按以下优先级决定取舍：

```
Priority(i) = HR(i) × IV(i) × Recency(i)
```

Priority 最低的 KU 优先进入 Archived 候选，经效用预演确认 ΔU ≥ 0 后执行归档。

## 二、演化机制：基于信息增益的状态转换

- **命中率**：该 KU 是否被召回
- **采纳率**：召回后是否被实际使用/引用

其中 α + β + γ = 1，权重由系统目标函数决定。

| 触发条件 | 执行操作 | 结果状态 |

## 三、反馈机制：净进化的数学保证

这是我方方案的核心，也是最难的部分。我方的答案分三层：

### 第一层：定义"进化"的数学语言

设知识库在时刻 t 的状态为 KB(t)，目标任务集合为 T，定义：

```
Fitness(KB, T) = E_{task ∈ T} [ Quality(KB, task) ]
```

**净进化的必要条件**：每次演化操作 op 执行后：

```
E[ Fitness(KB(t+1), T) ] ≥ Fitness(KB(t), T)
```

这个期望值，通过**历史反馈数据**估计，通过**A/B影子测试**在提交前验证。

### 第二层：演化操作的预提交验证

每次演化操作（Merge/Split/Archive）在正式写入前，必须通过一个**影子评估层**：

1. 在内存中构建操作后的 KB' 状态（不写磁盘）
2. 对最近 N 次历史查询重新跑一遍检索，对比 KB 与 KB' 的召回质量
3. 若 KB' 的加权质量评分 ≥ KB，则提交写入；否则操作撤销

这个机制保证：**每一次写入都是期望上的净进化，退化操作在提交前被拦截。**

### 第三层：资源受限下的优先级决策

当存储空间触及上限，必须 Archive 若干 KU 时，采用**边际贡献排序**：

```
MarginalValue(k) = Fitness(KB, T) - Fitness(KB \ {k}, T)
```

边际贡献最低的 KU 优先归档。

**关键约束**：若某 KU 的边际贡献为负（即删除它反而提升整体 Fitness），则立即 Archive，无需等待存储压力触发。

即用现有卡片库中语义相似卡片的历史效用加权平均，作为新卡片的先验估计。这不是空话——这是一个可执行的余弦相似度加权计算，数据来自已有卡片的效用历史向量。

### 问题一：演化机制

**每张卡片的核心元数据扩展如下：**


**状态转换的触发条件与判断标准：**

| 状态转换 | 触发条件 | 判断标准 |
|---|---|---|
| Created → Active | 卡片被首次创建 | 无需额外判断 |
| Active → Revised | 任务失败且溯源到此卡片 | 置信度下降超过阈值 θ_revise |
| Active → Merged | 两张卡片语义相似度 > σ，且合并后信息熵不增加 | JS散度 < δ_merge |

**执行流程（以Revised为例）：**
1. 任务执行失败 → 系统溯源：哪些卡片被调用
2. 对调用过的卡片执行贝叶斯更新：P(valid|failure) ∝ P(failure|valid) × P(valid)
3. 若置信度降至 θ_revise 以下 → 触发 Copy-on-Write，生成新卡片

### 问题二：反馈机制——如何数学保证净进化

这是我方方案最核心、最有力的部分。

**定义知识库的全局质量函数：**

```
Q(KB) = Σ_i [confidence_i × utility_i × recency_weight_i] / |Active Cards|
```

其中：
- confidence_i = 贝叶斯后验置信度
- utility_i = 卡片对高价值任务的覆盖率（按任务重要性加权）
- recency_weight_i = 时间衰减因子（防止过时知识占据高权重）

**净进化的数学保证：**

每次演化操作执行前，系统必须通过"演化沙盒"估算操作后的 ΔQ：

```
ΔQ = Q(KB after operation) - Q(KB before operation)
```

只有 ΔQ > 0 的操作才被允许提交。这是一个**贪心爬山保证**——每一步都不允许退化。

但贪心爬山有局部最优问题。我方的解法是：**引入周期性的模拟退火扰动**——每隔固定周期，允许 ΔQ 略微为负的操作（在受控范围内），以跳出局部最优。

**资源受限时的取舍逻辑：**

当存储达到上限，需要 Archive 某些卡片时，排序依据：

```
Priority_score = confidence × utility / storage_cost
```

Priority_score 最低的卡片优先归档。这确保单位存储空间的知识价值最大化。

**衡量知识库在变好还是变坏的核心指标：**

1. **任务成功率趋势**（Task Success Rate，TSR）：30天滚动窗口，必须呈上升趋势
2. **知识覆盖率**（Coverage）：能回答的任务类型占总任务类型的比例
3. **知识精度**（Precision）：被调用的卡片中，实际有效卡片的比例
4. **冗余率**（Redundancy Rate）：语义重叠卡片占总卡片的比例，应持续下降

## 二、我方对"冷启动问题"的完整闭环答案

对方的攻击集中于：新卡片没有历史数据，贝叶斯先验从哪来？

我方的完整答案分三层：

**第一层：先验来源是结构化的，不是语义模糊估计**

新卡片 $k_{new}$ 的先验效用 $U_{prior}(k_{new})$ 由以下三个可计算来源加权构成：

$$U_{prior}(k_{new}) = w_1 \cdot \bar{U}_{neighbor} + w_2 \cdot U_{topic} + w_3 \cdot U_{source}$$

- $\bar{U}_{neighbor}$：在嵌入空间中，$k_{new}$ 的K近邻卡片的历史效用均值（这是有历史数据的卡片）
- $U_{topic}$：该卡片所属主题类别的历史平均效用（类别级统计）
- $U_{source}$：信息来源的可靠性权重（如来自高质量文档的卡片先验更高）

**关键**：这个先验不是"内容越好效用越高"的直觉估计，而是**从已有效用历史中传播的统计量**。它的根基仍然是效用信号，只是经由邻域关系间接获取。

**第二层：先验的权重随观测数据严格衰减**

$$\alpha(t) = \frac{\alpha_0}{1 + \lambda \cdot n_{obs}(t)}$$

其中 $n_{obs}(t)$ 是卡片被实际查询并产生效用观测的次数。$\alpha_0=1$（冷启动时完全依赖先验），随每次真实查询积累，先验权重单调下降，观测权重单调上升。

这意味着：**新卡片在冷启动期处于"效用保护期"**——它不会在缺乏真实反馈时被驱逐，因为其效用估计的置信区间足够宽，系统不会轻易做出驱逐决策。

**第三层：驱逐决策需要置信度门槛，而非点估计比较**

MUR原则的完整版本：

$$\text{Accept } k_{new} \iff \hat{U}(k_{new}) - CI_{95\%}(k_{new}) > \hat{U}(k_{evict}) + CI_{95\%}(k_{evict})$$

即：只有当新卡片的效用**下置信区间**仍然高于被驱逐卡片的效用**上置信区间**时，才执行驱逐。

对于冷启动的新卡片，$CI_{95\%}$ 极宽，因此它几乎不会被接受（除非当前知识库中存在明确低效卡片）——这正是我们想要的保守行为。

**这三层合在一起构成完整闭环**：先验有明确来源 → 权重有明确衰减规则 → 决策有置信度约束。冷启动问题在这个框架内不是漏洞，而是被显式处理的边界条件。

$$U(k_{\text{new}}) > U(k_{\text{evict}}), \quad k_{\text{evict}} = \arg\min_{k \in K} U(k)$$

对新卡片缺乏历史数据的问题，引入贝叶斯效用估计：

$$\hat{U}(k_{\text{new}}) = \alpha \cdot U_{\text{prior}}(k_{\text{new}}) + (1-\alpha) \cdot U_{\text{observed}}(k_{\text{new}})$$

初始 $\alpha=1$（依赖语义先验），随观测数据积累衰减至 $0$。

**数学保证的逻辑**：在效用估计误差有界的温和假设下，MUR 保证每次置换操作后知识库期望效用 $\mathbb{E}_{q \sim T}[U(K, q)]$ 单调不降。这不是空话，而是一个可以被证伪的命题——如果估计误差超出界限，系统会通过效用监控检测到退化并回滚。

- **弱监督反馈**：检索后是否被引用、对话是否继续、任务是否成功结束
- **自监督反馈**：模型自评（用更强模型对检索结果打分）

这三条路径覆盖了从有人工标注到完全自主的全谱系场景。内容结构分析并非更"可操作"——它同样需要语义解析、矛盾检测等复杂管道，且产出的是一个与任务目标解耦的信号。

**争议四：演化的单调性 vs. 探索性**

**这个自我进化知识库系统，应该被设计成一个以不可变 Knowledge Unit 为基本单位、以候选变更包为演化通道、以统一目标函数和影子评估为准入裁判、以边际价值和预算约束为留并废依据的目标驱动演化系统——而不是一个以内部结构自洽为核心驱动的知识管理工具。**
