# Symposium 辩论结果

**议题：** 如何设计一个能够自我进化的知识库？
应用背景：GPU Kernel Design 的自动优化系统。

══════════════════════════════════════
背景说明
══════════════════════════════════════
用户原始代码（如 PyTorch）注重功能性，但在 GPU 上的实际性能往往未被充分优化。
GPU Kernel Design 的任务是：在保证正确性的前提下，不断发现并应用优化策略，
提升 Kernel 的运行速度（throughput / latency）。
这个过程会积累大量 Knowledge（优化策略、Trajectory、Experience），
这些 Knowledge 构成了需要自我进化的知识库。

══════════════════════════════════════
预设假设（不可更改，只可澄清适用边界）
══════════════════════════════════════
1. 存储与写入：文件系统存储，Copy-on-Write，Append-only
2. Knowledge Unit：唯一 UUID，卡片间可建立关联（Link）
3. 生命周期：Created / Revised / Merged / Split / Superseded / Archived
4. 全局 Canary Query Set（核心假定）：
   系统存在一个全局 Canary Query Set 作为衡量全局性能的基准。
   两个关键约束：
   (a) 运行 Canary 的代价较高——不能频繁触发；
   (b) 在 Canary 上 "move the needle"（产生可观测的显著变化），
       通常不是一两个局部 Knowledge Unit 的改变就能实现的，
       需要积累足够多的改变才能在 Canary 上显现出真实信号。
   因此，Canary 是长期 ground truth，而非短期 feedback 工具。
   （辩论中不质疑此假定，只可澄清其适用边界）

══════════════════════════════════════
在总知识量受限的情况下，如何设计反馈机制，
确保每一次进化操作是净进化而非净退化？

你必须正面回应以下两大核心挑战：

──────────────────────────────────────
【挑战一：Ablation Study 的成本与 delta 的真实性】
──────────────────────────────────────
针对少量 Knowledge 变化，消融实验（Ablation Study）是可行的；
但数量大时成本急剧上升。更关键的是：
并非所有小的 Knowledge 改变都能产生 real delta——
有些实验跑完后发现根本没有显著差异。

- 对"无效实验"（无 real delta），系统如何处理？如何从中学习？
- 做多少 Ablation 才足够？太少不可靠，太多耗不起。
- 在资源受限的情况下，如何设计成本与验证覆盖之间的权衡机制？

──────────────────────────────────────
【挑战二：Hypothesis 理解深度不足时的主动实验设计】
──────────────────────────────────────
有时系统只能得到一个速度数字，却不知道为什么快了。
真正的 Hypothesis（这个优化为什么有效）如果没被理解，
对未来系统提升帮助有限。

当现有经验深度不足时，系统能否主动设计新实验来
validate / verify Hypothesis，从而加深系统理解？

设计问题：
- 新实验可以是 sub-hypothesis，甚至递归产生孙 hypothesis
- 每次产生新实验都有成本（计算资源、时间、知识容量）
- 哪些实验是 no-regret 的（低成本、验证收益稳定）？
- 哪些是 costly 且收益不确定的（高成本、低期望 delta）？
- 如何设计系统使其能够：
    (a) 判断何时需要主动实验（知识深度不足的触发条件）
    (b) 评估实验的必要性（值不值得）与合理性（方向是否对）
    (c) 在资源受限下对实验优先级做出最优排序与权衡

## 第1轮：开篇立论

### 我方核心立场：以置信度驱动的假设图（Hypothesis Graph）作为进化引擎

---

## 一、立场定义

我方主张：**知识库的进化单位不是"性能数字"，而是"可证伪的假设"。**

每一个 Knowledge Unit（KU）本质上是一个带置信度的假设：

```
KU = {
  uuid,
  claim,
  evidence[],
  confidence: [0,1],
  scope,
  cost_to_verify
}
```

- **claim**：形如 `"在 M×N 维度 ≥ 128 的矩阵乘法中，使用 warp-level tiling + 共享内存 double buffering 可将 latency 降低 ≥ 15%"`
- **confidence**：贝叶斯后验 `P(claim | evidence)`，初始为先验
- **scope**：该假设适用的 GPU 架构、数据类型、shape 约束
- **cost_to_verify**：运行一次验证实验的预期计算代价

**进化 = confidence 的更新 + scope 的收缩/扩张 + 跨 KU 关系（蕴含、互斥、聚合）的显式化。**

---

## 二、判断标准

本场辩论应按以下标准评判优劣，按优先级排序：

1. **净进化保障**：方案能否在 Canary 不频繁运行的前提下，仍然防止知识库净退化？
2. **成本效率**：单位计算资源带来的知识置信度提升量（信息增益/成本）
3. **可解释性**：KU 能否解释"为什么快了"，而非仅记录"快了多少"
4. **可组合性**：多个局部 KU 的变化能否预测 Canary 上的全局 delta

---

## 三、两个核心论点

### 论点1：无显著 delta 的实验本身是一阶信号，必须被显式编码

挑战一的核心误区是把"无 real delta"的实验视为无用信息。我方的处理机制：

**null-result KU**：当 Ablation 实验显示 `|Δperf| < ε`（如 ε = 2%，低于测量噪声），系统生成一个 `Archived` 状态的 KU，其 claim 为否定命题：

> `"在 scope S 内，优化策略 X 对 metric M 无显著影响"`

这个 null-result 的价值在于：
- **剪枝搜索空间**：后续实验设计直接跳过该 scope×strategy 组合
- **更新先验**：降低同类策略在相似 scope 下的初始 confidence，减少后续重复实验
- **揭示交互效应**：若策略 X 单独无效，但与 Y 组合有效，null-result 精确标记了"单独无效"这个边界

**Ablation 数量的权衡**：不是固定跑多少，而是基于预期信息增益（Expected Information Gain, EIG）决策：

```
EIG(experiment e) = H(confidence_before) - E[H(confidence_after | result_e)]
```

当 `EIG < threshold_cost`（即预期信息增益低于实验代价折算值）时，停止该方向的 Ablation，优先执行 EIG 最高的候选实验。

---

### 论点2：主动实验设计必须基于假设图的"知识债"（Knowledge Debt）

挑战二的关键是：系统何时应该主动花成本去理解"为什么"？

**我方的触发条件**（三个必要之一即触发）：

1. **置信度悬空（floating confidence）**：某 KU 的 confidence ≥ 0.8，但其 evidence 全部来自单一 workload family——此时 scope 泛化风险高，必须主动在边界 workload 上验证
2. **图结构矛盾（graph conflict）**：两个 KU 的 claim 在逻辑上互斥，但各自 confidence 均 > 0.6——系统必须设计 decisive experiment 强制解决矛盾，而非让两者共存
3. **Canary surprise（预测失败）**：Canary 结果与 KU 组合预测值的偏差 > 2σ——触发溯因推理（abductive reasoning），逆向定位哪个 KU 的 scope 假设失效

**知识债的量化**：

```
KnowledgeDebt(KU) = confidence × (1 - scope_coverage) × dependency_fan_out
```

- `scope_coverage`：当前 evidence 覆盖的 scope 空间占声明 scope 的比例
- `dependency_fan_out`：依赖该 KU 的下游 KU 数量

系统优先偿还 KnowledgeDebt 最高的 KU，而非随机或轮询式实验。

我方要求本场辩论在以下赛道上展开：

|------|--------------|
| 进化单位 | 假设（可证伪命题）而非数据点 |
| 进化机制 | 贝叶斯置信度更新而非版本替换 |
| 实验策略 | EIG 驱动的主动学习而非穷举 |
| 失败处理 | null-result 显式编码而非丢弃 |
| 全局一致性 | Canary 作为假设图的压力测试而非唯一真相源 |

**一句话立场**：一个能自我进化的 GPU Kernel 优化知识库，其核心不是存储"什么优化有效"，而是维护一张活的假设图——每条边都有置信度，每次实验都在更新这张图，每次 Canary surprise 都在重构这张图的拓扑。

## 我方立场：以"假设驱动的置信度演化"为核心机制

---

### 一、框架先行：这场辩论应该比什么

在开篇，我要先锁定一件事：**这场辩论的核心争点不是"存什么"，而是"怎么知道自己进化了而非退化了"。**

知识库设计有两条路线之争：

- **路线A（被动积累派）**：触发即记录，靠 Canary 最终裁决，中间过程交给量变。
- **路线B（主动验证派）**：每一次知识状态转换，都必须经过显式的置信度更新，系统在写入新知识前，已经知道自己为什么写这张卡。

**我方坚守路线B。**

理由只有一个：在 GPU Kernel Design 这个场景里，**优化知识的错误代价极高**——一条错误的"规律"可能在数百个 Kernel 上被复用，且由于 Canary 触发代价高、信号滞后，等到 Canary 发现问题时，损害已经扩散。被动积累在这个场景下是危险的。

---

### 二、我方核心主张

**每一个 Knowledge Unit，都是一个带置信度的假设（Hypothesis），而非一个事实断言。**

具体而言，我方方案的骨架是：

#### 2.1 知识单元的基本结构

每张知识卡片，除了系统预设的 UUID / 生命周期状态外，必须携带三个额外字段：

```
Hypothesis:    "对于 memory-bound kernel，tiling size = 128 时 L2 hit rate 最优"
Confidence:    0.0 ~ 1.0（贝叶斯后验，初始值由提出来源决定）
Evidence_refs: [list of experiment UUIDs that update this confidence]
```

这三个字段，是我方方案的核心，也是区分"自我进化"与"自我膨胀"的分水岭。

#### 2.2 进化触发机制

知识卡片的状态转换，**不由时间或数量触发，而由置信度阈值触发**：

| 置信度区间 | 系统行为 |
|---|---|
| < 0.3 | 卡片处于 Hypothesis 态，不对外输出为建议 |
| 0.3 ~ 0.7 | 卡片处于 Tentative 态，可输出但附带不确定性标注 |
| > 0.7 | 卡片晋升为 Established，可作为 prior 影响后续实验设计 |
| 新证据与现有卡片冲突 | 触发 Revised（CoW 机制），旧卡片变为 Superseded |
| 两张卡片置信度均高且语义互补 | 触发 Merged，生成新卡片，置信度重新计算 |

**关键点：** 系统永远不直接"修改"一个知识单元的置信度字段——它通过追加新的 Evidence_ref、产生新卡片来演化。这与预设假设的 Append-only / CoW 机制完全一致。

#### 2.3 反馈机制的核心设计

这是本场辩论最重要的问题，我方在第一轮就给出核心答案，而不是留到后面打太极：

**反馈分两层，各司其职：**

**层一：局部快速反馈（Low-cost, High-frequency）**
- 每次 Kernel 跑完，记录 `(optimization_applied, context_features, delta_perf)`
- 这个三元组作为 Evidence，用贝叶斯更新对应 Knowledge Unit 的置信度
- 代价：O(1) per experiment，无需额外跑 Ablation

**层二：Canary 作为全局校准（High-cost, Low-frequency）**
- Canary 不用于评价单个 Knowledge Unit，而是用于检测**知识库整体是否发散**
- 具体机制：Canary 触发时，比较当前 Established 知识集合 vs. 上次 Canary 时的集合
- 如果 Canary 分数下降，触发"知识回溯"：将近期晋升为 Established 的卡片批量降级回 Tentative，等待重新积累证据

这个设计的精髓在于：**Canary 不是裁判员，是熔断器。** 它不评价对错，只触发保护机制。

---

### 三、对两大核心挑战的预先定位

**挑战一（Ablation 成本）：**
我方方案不依赖传统 Ablation Study 作为主要验证手段。置信度的更新来自日常实验的三元组记录，Ablation 只在置信度长期停滞在 0.3~0.7 区间时才触发，作为"强制解锁"手段——这将 Ablation 的使用频率压缩到最低必要集合。

**挑战二（主动实验设计）：**
当一张卡片的假设深度不足（即我们只知道"快了"但不知道"为什么"）时，系统会自动生成 **Diagnostic Experiment**：通过控制变量，将一个"黑盒 delta"分解为可归因的 sub-hypotheses，每个 sub-hypothesis 对应一张新的 Created 卡片。这个过程是 no-regret 的，因为无论结果如何，系统都获得了可以更新置信度的新证据。

---

### 四、我方赛道宣言

**这场辩论，应该按以下标准来比：**

1. **知识进化是否有显式的可验证路径**（不能只靠"量变产生质变"）
2. **在 Canary 触发代价高的约束下，局部反馈机制能否独立保证单次进化的净正性**
3. **系统对"为什么快了"的理解深度，而非仅仅"快了多少"**

# 第2轮：聚焦质询

对方立论与我方在框架层面高度相似——都采用假设驱动、置信度更新、两层反馈。这不是巧合，而是恰好说明这个方向是正确的。但正因如此，我必须把质询聚焦在对方方案中真正存在漏洞的**具体机制**，而非框架之争。

---

## 质询问题一：EIG 计算本身依赖一个无法自举的先验——你的框架在冷启动时是循环定义的

对方提出，实验是否执行取决于：
> EIG(experiment e) = H(confidence_before) - E[H(confidence_after | result_e)]
> 当 EIG < threshold_cost 时，停止该方向的 Ablation。

**我的质询：**

EIG 的计算需要 `E[H(confidence_after | result_e)]`，即**预测"实验结果出来后，置信度分布会变成什么样"的期望**。这个期望值从哪里来？

它必须来自一个**似然模型** `P(result_e | claim)`——也就是说，你需要事先知道"如果 claim 为真，这个实验大概率会出什么结果"。

这正是知识库本身要存储和积累的东西。

**这是循环依赖**：
- 用 EIG 决定做哪些实验 → 需要似然模型 → 似然模型来自已有 KU → 但 KU 的质量正是需要通过实验来建立的

在知识库稀疏的**冷启动阶段**，似然模型要么缺失，要么基于极少量证据——此时 EIG 的计算是不可靠的，但对方却将 EIG 作为**最核心的实验调度决策依据**。

**请对方明确回答：**
1. 冷启动阶段（KU 数量 < N，N 如何定义？）如何获得可靠的似然估计？
2. 如果用平坦先验（uniform prior）代替，EIG 对所有候选实验几乎相等，调度机制退化为随机选择——对方如何避免这一退化？

**为什么这是薄弱点：** EIG 驱动的实验调度是对方论点2的核心引擎。如果这个引擎在最需要它的阶段（知识库初建期）无法可靠运转，那整个"主动实验设计基于假设图知识债"的论点就只在知识库已经足够成熟时才成立——而那恰恰是最不需要主动设计的时候。

---

## 质询问题二：`scope` 的收缩与扩张，隐含了一个未定义的距离度量

对方将"进化"定义为：
> 进化 = confidence 的更新 + **scope 的收缩/扩张** + 跨 KU 关系的显式化

对方还提出触发条件：
> floating confidence：某 KU 的 confidence ≥ 0.8，但 evidence 全部来自同一 GPU 架构 → scope 过窄，需要 scope 扩展实验

**我的质询：**

`scope` 被定义为 `{GPU 架构, 数据类型, shape 约束}`，这是一个**多维离散+连续混合空间**。

当系统判断"scope 过窄，需要扩张"时，它必须回答：**扩张到哪里？**

- 从 A100 扩到 H100，还是扩到 RTX 4090？
- 从 FP16 扩到 BF16，还是扩到 FP8？
- shape 约束从 `M×N ≥ 128` 扩到 `≥ 64` 还是 `≥ 256`？

这些选择需要一个 **scope 空间上的邻近性度量**（proximity metric）——否则"扩张"是无定向的，系统无法自动确定下一个扩展实验的目标 scope。

对方的 KU 结构中没有定义这个度量。`scope` 只是一个字段，不是一个带拓扑结构的空间。
